# Learnings log: 2026-03
## 2026-03-13
- The repository began effectively empty, so the first useful checkpoint had to prioritize scaffolding, documentation, and validation loops over product polish.
- Feed-first architecture is the safest initial Giant Bomb strategy because legacy structured APIs are not currently available.
- The first foundation loop is more maintainable when feature screens stay UI-focused and the app shell owns repository wiring.
- A lightweight manual probe script is enough for the first live integration seam; full automated live network validation would add unnecessary fragility this early.
- `verifyFast` and `verifyLocal` establish an explicit Ralph loop cadence for future human and agent contributors.
- AGP 9's built-in Kotlin support means Android modules should not also apply `org.jetbrains.kotlin.android`.
- The managed-device smoke lane validated successfully with a `pixel6Api36` device once the Google APIs x86_64 system image was provisioned, but AGP still emits an ABI migration warning during setup.
- A production-shaped repository can still be introduced before live credentials are available by combining persisted non-sensitive config, session-only password handling, and seeded fallback content.
- Parsing RSS feeds needs at least `media:content`, `enclosure`, and `link` fallback handling to avoid overfitting to a single feed shape.
- A useful playback slice can land before end-to-end live validation by wiring Media3 for any available playback URL and showing a clear fallback when seeded or incomplete items are not yet playable.
- The user-facing auth surface should present the only supported login path as “Sign in to Giant Bomb Premium” instead of exposing internal integration strategy choices.
- Copied Giant Bomb Premium feed URLs should be treated as first-class inputs; username and password need to remain optional fallback fields instead of hard requirements.
## 2026-03-16
- Once the app had both a flat Library view and a grouped Shows view, the split stopped paying for itself; the better UX was a single browse surface with grouping, sorting, filtering, rich cards, and jump navigation layered together.
- A right-edge jump rail works better when the rail itself stays visually slim and persistent while the labels behave like transient flyouts instead of a permanently space-hungry side panel.
- Jump rail labels are much easier to scan when they are single-line, slightly longer, and padded like compact pills instead of multi-line blocks.
- Giant Bomb-inspired polish in this app works best when the hierarchy stays content-first: stronger media art, slimmer navigation chrome, and warmer accent colors without trying to clone the website literally.
- Future agent loops need the README/spec/architecture docs refreshed after major product-shape changes; otherwise the docs start pulling contributors back toward already-completed “foundation” work.
- Roborazzi is a better first screenshot-regression fit for this repo than a separate preview-only renderer because it layers onto the existing Android app module and Gradle validation flow with less architectural churn.
- Screenshot baselines need to live under source control instead of `build/` output; otherwise clean builds erase the very history needed for meaningful visual regression checks.
- Screenshot coverage expands fastest when tests target deterministic product states first; Auth and Player fallback/preparing surfaces are worth locking down before trying to snapshot a live `PlayerView`.
- Grouped browsing benefits from canonicalizing common Giant Bomb abbreviations early; mapping `UPF` to `Unprofessional Fridays` at the model/parser layer is cheaper than trying to reconcile split group labels in the UI later.
- AndroidX Security Crypto is no longer a clean drop-in future-proof choice for this use case, so an app-owned Android Keystore AES/GCM wrapper is a better fit for protecting the copied feed URL at rest without tying repository code to deprecated preference APIs.
- Once feed-first auth became the validated short-term path, the best repeat-use improvement was not a speculative browser session spike but an explicit remembered-password option built on the same encrypted-at-rest store.
- `Clear synced data` and `Forget saved credentials` are meaningfully different actions in a feed-backed app; separating them improves repeat-use ergonomics without weakening the security story.
- If secure persistence falls back to plaintext legacy storage, saved credentials should degrade toward session-only behavior rather than ever writing a password outside the encrypted payload.
- The manual Giant Bomb probe should follow the same auth model as the app itself: copied feed URL required, basic-auth credentials optional, and probe output limited to structural metadata instead of raw authenticated XML previews.
