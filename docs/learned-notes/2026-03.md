# Learnings log: 2026-03
## 2026-03-13
- The repository began effectively empty, so the first useful checkpoint had to prioritize scaffolding, documentation, and validation loops over product polish.
- Feed-first architecture is the safest initial Giant Bomb strategy because legacy structured APIs are not currently available.
- The first foundation loop is more maintainable when feature screens stay UI-focused and the app shell owns repository wiring.
- A lightweight manual probe script is enough for the first live integration seam; full automated live network validation would add unnecessary fragility this early.
- `verifyFast` and `verifyLocal` establish an explicit Ralph loop cadence for future human and agent contributors.
- AGP 9's built-in Kotlin support means Android modules should not also apply `org.jetbrains.kotlin.android`.
- The managed-device smoke lane validated successfully with a `pixel6Api36` device once the Google APIs x86_64 system image was provisioned, but AGP still emits an ABI migration warning during setup.
- A production-shaped repository can still be introduced before live credentials are available by combining persisted non-sensitive config and seeded fallback content.
- Parsing RSS feeds needs at least `media:content`, `enclosure`, and `link` fallback handling to avoid overfitting to a single feed shape.
- A useful playback slice can land before end-to-end live validation by wiring Media3 for any available playback URL and showing a clear fallback when seeded or incomplete items are not yet playable.
- The user-facing auth surface should present the only supported login path as “Sign in to Giant Bomb Premium” instead of exposing internal integration strategy choices.
- Copied Giant Bomb Premium feed URLs should be treated as first-class inputs.
## 2026-03-16
- Once the app had both a flat Library view and a grouped Shows view, the split stopped paying for itself; the better UX was a single browse surface with grouping, sorting, filtering, rich cards, and jump navigation layered together.
- A right-edge jump rail works better when the rail itself stays visually slim and persistent while the labels behave like transient flyouts instead of a permanently space-hungry side panel.
- Jump rail labels are much easier to scan when they are single-line, slightly longer, and padded like compact pills instead of multi-line blocks.
- Giant Bomb-inspired polish in this app works best when the hierarchy stays content-first: stronger media art, slimmer navigation chrome, and warmer accent colors without trying to clone the website literally.
- Future agent loops need the README/spec/architecture docs refreshed after major product-shape changes; otherwise the docs start pulling contributors back toward already-completed “foundation” work.
- Roborazzi is a better first screenshot-regression fit for this repo than a separate preview-only renderer because it layers onto the existing Android app module and Gradle validation flow with less architectural churn.
- Screenshot baselines need to live under source control instead of `build/` output; otherwise clean builds erase the very history needed for meaningful visual regression checks.
- Screenshot coverage expands fastest when tests target deterministic product states first; Auth and Player empty/preparing surfaces are worth locking down before trying to snapshot a live `PlayerView`.
- Grouped browsing benefits from canonicalizing common Giant Bomb abbreviations early; mapping `UPF` to `Unprofessional Fridays` at the model/parser layer is cheaper than trying to reconcile split group labels in the UI later.
- AndroidX Security Crypto is no longer a clean drop-in future-proof choice for this use case, so an app-owned Android Keystore AES/GCM wrapper is a better fit for protecting the copied feed URL at rest without tying repository code to deprecated preference APIs.
- Once feed-first access became the validated short-term path, the safest repeat-use improvement was persisting only the copied feed URL in the encrypted-at-rest store.
- `Clear synced data` should preserve the saved feed URL so repeat syncs stay cheap without reintroducing broader auth state.
- If secure persistence falls back to plaintext legacy storage, it should store only the feed URL and never grow back into broader credential persistence.
- The manual Giant Bomb probe should follow the same model as the app itself: copied feed URL required and probe output limited to structural metadata instead of raw authenticated XML previews.
- Once the product question shifted from "can we sign in somehow?" to "what can we support responsibly?", the cleanest architectural boundary was to stay feed-first and explicitly avoid browser-login or scraping-style automation without an official Giant Bomb path.
- The next useful live-validation tool was a narrow feed probe, not a broader auth experiment: validate the copied feed URL directly so the product decision can be grounded in observed behavior instead of assumptions.
- After `SessionCookie` support was removed, keeping a standalone `AuthMode` enum no longer represented a real product choice; deleting it made the feed-sync flow and persistence model easier to reason about.
- A real Giant Bomb `premium-videos` 1080p feed validated the current feed-first assumption: on March 16, 2026, `feed-url-only` returned HTTP 200 with 84 items, 84 `media:content` tags, 84 `enclosure` tags, and 84 item links.
- The app's playback path is even simpler than the feed sync path: once a feed item is selected, the current code hands `playbackUrl` directly to Media3 without cookies or extra auth headers, so the right validation seam is the extracted media URL itself.
- A live sample of the first three items from the same `premium-videos` 1080p feed returned direct `206 video/mp4` responses from `cdn.jwplayer.com` with no query string, which is strong evidence that the validated feed path already exposes directly playable media targets instead of requiring browser-session replay.
- Once both live feed probes and live playback probes succeeded with the copied feed URL alone, the next useful simplification was to delete the fallback credential branch entirely so the product language, repository model, and validation flow all matched the supported behavior.
## 2026-03-24
- The background-refresh job was still present on a connected device even after the user reported only seeing one notification, which shifted the likely failure mode away from “scheduler missing” and toward “repeat delivery not clearly surfaced.”
- Reusing one fixed notification ID for all new-release alerts is too weak for repeat-use reliability because later detections can collapse into the same record instead of leaving clearly distinct evidence in the tray.
- Notification behavior needs explicit product targets, not just implementation code: what should stay silent, what should alert, and what should be visible on-device all need to be spelled out and regression-tested.
## 2026-03-25
- Giant Bomb site-content heuristics become much easier to maintain when feed-parser inference, grouping logic, and UI-facing title/cast/topic derivation all share one dedicated module instead of splitting rule tables between `core:model` and `core:network`.
## 2026-03-29
- For LookTube's current Android constraints, a controlled `whisper.cpp` integration is a better local-caption fit than the Android Moonshine path that currently appears mismatched on minimum SDK and ABI support.
- The local caption fallback should be treated as the foundational path, not a premium add-on behind a future provider configuration.
- Feed-backed direct `.mp4` playback URLs are useful twice: they keep the playback path simple and also make sender-side audio extraction feasible for offline caption generation.
- Generated captions need to remain separate from feed-derived metadata; storing them as per-video WebVTT sidecars keeps the feed contract clean while still making the tracks reusable for local playback and cast.
- Default Media3 cast conversion is too implicit for locally generated subtitles; explicit Cast text-track mapping plus a lightweight sender-hosted HTTP server is the safer contract for delivering sidecar captions to receivers.
- Screenshot coverage stays more maintainable when new baselines lock down deterministic feed-sync and playback-status states before trying to snapshot the live `PlayerView` path.
## 2026-03-30
- First-open polish is stronger when the product shows an intentional empty-library state instead of clearly fake seeded content; placeholder media makes the app feel broken before sync has even been attempted.
- A very short launch intro can add some brand presence without hurting usability, but only if it is cold-start-only, auto-dismisses quickly, and yields instantly to any tap.
- `whisper.cpp` behaves best when the app's outer transcription chunking stays aligned with its natural ~30 second decode windows; stretching chunks far beyond that can make on-device wall time worse even after removing a tiny-chunk regression.
- If a higher-spec build already ships a faster local caption engine, that flavor should boot into the faster engine by default instead of making users rediscover the slower compatibility fallback after every launch.
- `whisper.cpp` does not provide a single trustworthy mobile speed promise; the more useful product contract is surfacing measured on-device throughput plus the latest chunk wall time and native encode/decode timings so future tuning is guided by observed runtime instead of guesswork.
## 2026-03-31
- The safest repeat-use auto-caption feature is a persistent Settings toggle that only targets newly discovered playable videos and reuses the exact same local caption pipeline as manual generation instead of inventing a second caption path.
- Small shell and browse affordances matter more than bigger chrome changes: a compact top-bar playing pill, fully tappable group headers, and clearly different collapsed versus expanded states improve comprehension without adding new navigation.
